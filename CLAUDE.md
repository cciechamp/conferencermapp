# Conference Room Booking System

## Project Overview
A small Flask REST API for reserving shared conference rooms across office
buildings. It manages three entities — **conference rooms**, **employees**
(organizers), and **bookings** — and its core job is to let an employee book a
room for a time window while **preventing double-bookings**: every create and
reschedule runs an overlap check so two scheduled meetings can never share the
same room and time. It is a backend/JSON service (no UI) intended to be
consumed by internal tools, scripts, or a future frontend.

The app follows the Flask **app-factory + blueprints** pattern. `create_app()`
in `app.py` wires up the SQLite database, registers the `rooms` and `bookings`
blueprints, and auto-creates tables on startup. Routes are thin HTTP handlers;
serialization lives on the models (`to_dict()`), and the double-booking rule
lives in `utils/conflict.py`.

### Layout
```
app.py              # app factory, /health, blueprint registration
models.py           # ConferenceRoom, Employee, Booking (SQLAlchemy models)
routes/rooms.py     # /rooms endpoints (list, get, availability, available)
routes/bookings.py  # /bookings endpoints (list, get, create, reschedule, cancel)
utils/conflict.py   # check_overlap() — the double-booking guard
db/seed_data.py     # drops + recreates tables and loads sample data
db/bookings.db      # SQLite database (gitignored, regenerated by seed)
tests/              # pytest scaffolding (pytest + pytest-cov are installed)
```

### Data model
- **conference_rooms** — `id`, `name`, `capacity`, `location`
- **employees** — `id`, `name`, `email` (unique), `department`
- **bookings** — `id`, `room_id` → rooms, `organizer_id` → employees,
  `start_time`, `end_time`, `meeting_title`, `attendees`, `status`,
  `created_at`. `status` is `scheduled` or `cancelled`.

Every JSON response uses one envelope: `{"data": ..., "error": ..., "status": <int>}`.

### Running it
```bash
pip install -r requirements.txt
python db/seed_data.py     # optional: load 5 rooms, 10 employees, 20 bookings
python app.py              # serves on http://localhost:5000 (debug=True)
```

### Usage examples
```bash
# Health check
curl http://localhost:5000/health

# List all rooms
curl http://localhost:5000/rooms

# Bookings for room 1 (optionally filter by organizer)
curl "http://localhost:5000/bookings?room_id=1&organizer_id=1"

# Free 30-min slots for room 1 on a given day (date is REQUIRED)
curl "http://localhost:5000/rooms/1/available?date=2025-07-01"

# Create a booking (409 if the slot conflicts, 400 on bad input)
curl -X POST http://localhost:5000/bookings \
  -H "Content-Type: application/json" \
  -d '{"room_id": 1, "organizer_id": 1, "start_time": "2025-08-01T10:00:00", "end_time": "2025-08-01T10:30:00", "meeting_title": "Design review", "attendees": 4}'

# Reschedule (PUT) — changes only start/end time
curl -X PUT http://localhost:5000/bookings/1 \
  -H "Content-Type: application/json" \
  -d '{"start_time": "2025-08-01T14:00:00", "end_time": "2025-08-01T14:30:00"}'

# Cancel (soft delete — sets status to "cancelled")
curl -X DELETE http://localhost:5000/bookings/1
```

## Tech Stack
- Language: Python 3.11

- ORM: Flask-SQLAlchemy 3.1
- Database: SQLite (db/bookings.db)
- Testing: pytest 8 + pytest-cov (see `tests/`)

## Coding Conventions
1. **Uniform response envelope.** Every route returns
   `{"data": ..., "error": ..., "status": <int>}`. On success `error` is
   `None`; on failure `data` is `None` and a human-readable `error` string is
   set. The HTTP status code is set explicitly to match `status` for
   non-200 responses (e.g. `return jsonify({...}), 404`). Keep this shape for
   any new endpoint.
2. **App factory + blueprints.** New route groups go in `routes/<name>.py` as a
   blueprint named `<name>_bp` and are registered in `create_app()`. Blueprints
   are mounted at the **root** (no `url_prefix`), so paths are literal
   (`/rooms`, `/bookings`).
3. **Layering.** Routes handle HTTP concerns and input validation only; model
   serialization belongs in each model's `to_dict()`; reusable business logic
   (like overlap checking) belongs in `utils/`. Don't inline booking-conflict
   logic into a route — call `check_overlap()`.
4. **Validate before writing.** Datetimes are accepted as ISO 8601 and parsed
   with `datetime.fromisoformat`; `end_time` must be strictly after
   `start_time`. All validation (missing fields, bad dates, conflicts) happens
   and returns early *before* any `db.session.commit()`.
5. **Naming.** snake_case functions and variables; plural resource paths
   (`/rooms`, `/bookings`); path params typed in the rule (`<int:room_id>`);
   module-level tunables in UPPER_CASE (`BUSINESS_START`, `SLOT_MINUTES`).
6. **Soft deletes.** Cancelling a booking sets `status = "cancelled"` rather
   than deleting the row. Availability and conflict checks filter on
   `status == "scheduled"`, so cancelled rows are ignored but preserved.
7. **Google-style docstrings.** Public methods and routes document purpose,
   Args (with types), Returns/error envelopes, examples, and Browser + cURL
   invocations (see existing routes for the template).

## Do Not Touch
- **`utils/conflict.py` — the strict `<` / `>` comparisons in `check_overlap`.**
  The overlap test uses `Booking.start_time < end_time` and
  `Booking.end_time > start_time` (strict, not `<=`/`>=`) *on purpose* so that
  back-to-back bookings (e.g. 09:00–09:30 followed by 09:30–10:00) are allowed.
  Loosening these to `<=`/`>=` would reject valid adjacent bookings and is a
  silent behavior change to the system's core rule. Don't change without a
  deliberate reason and matching tests.

## Useful Context
- **Two room endpoints with near-identical names do opposite things.**
  `GET /rooms/<id>/availability` returns the room's **booked** (scheduled)
  slots; `GET /rooms/<id>/available` returns the **free** 30-minute slots on a
  09:00–18:00 grid. `availability` accepts an *optional* `date`; `available`
  **requires** `date` (returns 400 without it). Easy to confuse — check which
  one you mean.
- **Seed bookings land in the evening, not during business hours.** In
  `db/seed_data.py`, the start time is computed as
  `base(09:00) + timedelta(hours=9 + (idx % 4))`, so the 20 "Team Sync"
  bookings fall at 18:00–21:30. Because `/rooms/<id>/available` only grids
  09:00–18:00, querying availability for a seeded date returns a *fully free*
  day, which looks like a bug but is just the seed data sitting outside the
  window. To test slot exclusion, book an in-hours slot first. Note also that
  `seed_data.py` calls `db.drop_all()` — **running it wipes the database.**
